# Change Policy

Use this file when a request changes scope, sequencing, or shared contracts.

## Doc update order

1. PRD
2. UX flow
3. Frontend and backend implementation docs
4. AI safety if wording or routing changes
5. Analytics, QA, and sprint docs

## Schedule guardrails

- Protect the core identity first.
- Cut visual polish before cutting core modules.
- Cut optional card variety before cutting safety.
- Cut nice-to-have transitions before cutting data contracts.

## When to escalate

Escalate if a request would:

- turn mode scenes into games
- add a new primary module on Home
- require a new service or database
- change the safety-block behavior
- create content ops overhead for one person

## Acceptance rule

A scope change is acceptable only if:

- it fits the original-demand-compatible MVP
- it does not break the 8-week main-build target too badly
- the affected docs can be kept aligned in one pass
